## 2. 逐条核查结论（证据行号）

### 核查 1 — Requirement 1：不越写边界 → PASS

- `useRescheduleEvent.ts:52` `mutationFn: ({eventId,startAt,endAt}) => updateEvent(eventId, {startAt, endAt})`。
  payload **只含 `startAt`/`endAt`**，无 title/notes/status/category/targetId 夹带。
- `updateEvent`（`plansMutations.ts:73-83`）打 `PATCH /plans/events/{eventId}` + `PlanEventUpdateRequestV2`，
  即 SIK-138 既有端点 + 既有 DTO。DTO（`api.generated.ts:4007-4026`）所有字段 optional，`{startAt,endAt}` 子集合法。
- import 路径 `@sikao/api-client/plansQueries`（hook:33-34）—— `plansQueries.ts:108` 确实 re-export 了 `updateEvent`，路径有效，非野路径。
- 无新增后端端点。

H1 越界判定（题目点名）：**认同 Runner 的判断，不构成越界。** 理由：
`useUpdateEvent`（`plansMutations.ts:216-227`）在 hook-call 时绑定固定 `eventId`，
而 drop 的 target id 到 drop 一刻才知道，无法在渲染期用 `useUpdateEvent`。
新 hook 复用的是**同一 `updateEvent` 函数 / 同端点 / 同 DTO**，仅在 app 层包一层 `useMutation`
以接收 per-call `eventId` 变量 —— 网络边界 0 变化，不是新跨服务契约。归类为 Finding L-1（DRY，见 §3），非 H1 阻塞。

### 核查 2 — Requirement 4：落点 + 时间保持 → PASS

- 整日平移：`rescheduleEvent.ts:73` `deltaMinutes = dayDelta(fromDay,toDay) * MINUTES_PER_DAY`，
  `dayDelta`（:38-47）按 UTC 00:00 取整日差，`Math.round` 防浮点。`addMinutesToIso` 加整日分钟数。
- 时分/时长/tz 保持：`addMinutesToIso`（`timezone.ts:38-41`）= 在 epoch 上加固定毫秒。
  加 N×1440min 是整日偏移，**同一墙钟时刻在固定 offset 时区下保持不变**，时长（end-start）天然不变。
  单测证实：`rescheduleEvent.test.ts:24-31`（forward 保 08:00/09:30）、:39-44（跨月）、:46-54（跨日事件 22:00→次日 01:00 平移后仍跨日）。
- 同日 drop noop 不发请求：`resolveCalendarDrop.ts:68-70` `overId === data.fromDay → {kind:'noop'}`；
  `MonthGridDnd.tsx:213-215` `decision.kind !== 'reschedule'` 直接 return，不调 mutate。单测 `resolveCalendarDrop.test.ts:32-34`。
- 跨日按 fromDay→toDay delta 平移：契约的"grab the slice you see"语义，`fromDay = slice.day`（`DraggableChip` data:99 `fromDay: item.slice.day`），正确。

注（非阻塞，见 Finding L-2）：`addMinutesToIso` 经 `toISOString()` 会把 `+08:00` 表示归一成 `Z`。
**同一时刻**，对单 tz app 无语义影响；但持久化表示从带 offset 变 UTC，值得在多 tz 引入前留意（代码注释 `rescheduleEvent.ts:23-31` 已显式声明此限制，诚实）。

### 核查 3 — Requirement 6 / H7：Fail-Fast（重点）→ PASS

逐点位对 Requirement 6 清单：

- **drop 到非 cell（over==null）→ cancel 无副作用**：`MonthGridDnd.tsx:203` `overId = event.over==null ? null : String(...)`；
  `resolveCalendarDrop.ts:65-67` `overId==null || data==null → {kind:'cancel'}`；:213-215 return，无 store 写、无请求。
  单测 `resolveCalendarDrop.test.ts:23-29`（cancel outside / 无 data）。**禁猜最近日期** —— 确认无最近格 fallback。
- **时间计算异常 → 抛错不 `?? 原值`**：`rescheduleEvent.ts:64-71` 对 day stamp 正则 + event 时间 `Date.parse` NaN 双重显式 `throw`；
  `dayDelta:43-45` 越界日期再 throw。**无 `?? 原值` 兜底**。单测 :56-62（malformed throw）。
- **malformed 先 toast 再 throw（非 silent）**：`MonthGridDnd.tsx:205-211` `try{resolve}catch{toast.error(...); throw}`。
  顺序正确：**先 toast 后 throw**，异常不被吞，且此时**尚未 `upsertOptimisticEvent`**（乐观写在 :219，throw 发生在 :210 之前），无残留。
- **PATCH reject → removeOptimisticEvent 回滚 + 显式 toast**：`MonthGridDnd.tsx:227-231` `onError` 先 `removeOptimisticEvent(eventId)` 再 `toast.error`。乐观 patch 不残留。
- **无 silent catch / 吞异常**：全文件仅一处 catch（:207），且 re-throw + toast；mutation 用 `onSuccess`/`onError` 回调显式分流，无 `.catch(()=>{})`。

结论：H7 合规，5 个点位全覆盖。`resolveCalendarDrop.ts` 头注（:36-43）明确把"malformed 由调用点 toast 接住、resolver 保持纯"的职责边界写清，符合设计意图。

### 核查 4 — 乐观一致性 → PASS（含 1 个 L 级备注）

- **eventId 是真实 event.id**：`DraggableChip` data:97 `eventId: item.event.id`（真实持久 id，mutation target），
  与 draggable 的 handle id `entryId = ${occurrenceRef}|${day}`（per-slice，:95/:268）**显式区分**。
  `upsertOptimisticEvent(eventId,...)`（:219）+ `mutate({eventId,...})`（:220）用的都是真实 id。
- **成功后 removeOptimisticEvent 让 refetch 接管**：`:224-225` `onSuccess → removeOptimisticEvent(eventId)`，
  且 `useRescheduleEvent.ts:53-58` `onSuccess` invalidate plans/dashboard/progress/recommendations 触发 refetch。
  乐观值不被当真值留存 —— 符合 Requirement 2「不把乐观值当真值」。
- **回滚 vs 成功竞争（双清）**：react-query 单次 mutation 的 `onSuccess`/`onError` 互斥，只触发一个；
  `removeOptimisticEvent`（`usePlanStore.ts:62-67`）是幂等 `Map.delete`，即便重复也无副作用。无竞争风险。

L 级备注（Finding L-3）：乐观写入的是 `{startAt,endAt}`，但**月视图 chip 不渲染时间字段**
（`MonthEventChip` 渲染 kind/title/category/status/source/link/target，无 startAt），
且 chip 落格由 `eventsByDay`（`MonthCalendarView.tsx:226-230`，**仅基于 `query.data`**）决定，
不经 optimisticEvents 重投影。因此本 Wave 的乐观 `{startAt,endAt}` patch **视觉上不会让 chip 移动到新格**，
要到 PATCH 成功 refetch 后 chip 才落到新日。

- 这与 Requirement 2「chip 渲染层在 occurrence 投影前 merge」存在张力：当前 merge 发生在 chip 表面（投影后），不在投影前。
- 不构成 H7/写边界缺陷，回滚/一致性逻辑本身正确；但**契约 §0 Wave 2 行「drop-target 高亮 + 乐观预览」中的"乐观预览"在月视图无可见效果**。
- 判为 L（诚实性/契约措辞），见 §3 建议。功能正确性不受影响（成功路径 refetch 落位、失败路径回滚均 OK）。
