## Round 1 Findings

1. `high` requirements / plan / design
   - 问题：`Requirement 9: Fail-Fast` 把 event/session 缺失和 unsupported/no-graded 也写成了整批 fail-fast，但另外三份文档又把它们定义成 per-item `availability`。
   - 处理：收口为“request-level 4xx，item-level availability”，并保留显式 empty-state 渲染能力。

2. `high` plan / design
   - 问题：一边要求 response 按请求顺序返回，一边又让 query key 用排序后的 `eventIds`，会造成同集合不同顺序的缓存/响应错配。
   - 处理：锁定前端发送前 `dedupe + stable sort(eventIds)`，response 保持输入顺序，UI 按 `eventId` 建索引消费。

3. `medium` visual contract
   - 问题：empty-state labels 少了 `session_not_found` / `no_graded_items` 的显式文案，且 H11 token map / acceptance hooks 不够具体。
   - 处理：补齐 `关联失效`、`无判题数据`、`event_unavailable` 等文案，并把 token map 改成 prototype/current var -> V5 token 映射，补上 dual-open diff hook。

4. `medium` contract gap
   - 问题：event-level missing/hidden 被降成 per-item availability 后，枚举里没有对应状态。
   - 处理：新增 `event_unavailable`，并同步到 plan / requirements / design / visual contract。
