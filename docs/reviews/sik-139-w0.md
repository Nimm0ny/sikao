## 2. 逐条核查结论

### 2.1 契约正确性（锚点职责）— PASS

- `data-event-id` = `event.id`（`MonthEventChip.tsx:161`）。`event.id` 经 `calendarEvents.ts`
  确认是真实持久化事件 id（`eventToCalendarInput` 用 `event.id` 作 `sourceId`），类型
  `PlanEventReadV2.id: string`（`api.generated.ts:3970`）。✅ 是 mutation target，非 per-slice id。
- `data-peek-anchor` = `peekAnchorId ?? event.id`（`MonthEventChip.tsx:162`）。View 传入
  `entryId = ${occurrenceRef}|${day}`（`MonthCalendarView.tsx:163`）。`occurrenceRef =
  ${event.id}:${dateStamp}`（`calendar-engine/src/recurrence.ts:12`），故 anchor 形如
  `e1:2026-05-27|2026-05-27` —— per-slice 唯一，内嵌真实 id 但与 id 不同。缺省回退 event.id。✅
- 两者职责清晰未混用：mutation 走 `data-event-id`（真实 id），拖拽/peek 定位走
  `data-peek-anchor`（slice 锚）。这是 Phase 3 的边界基石，方向正确。✅

### 2.2 乐观 merge（D20 / H7）— PASS

- merge 发生在 **chip 层**：`MonthEventChip.tsx:138` `const event = optimisticPatch ?
  { ...sourceEvent, ...optimisticPatch } : sourceEvent;`。符合 design State Flow
  「chip 渲染层在 occurrence 投影前 merge」。✅
- read-only：View 仅 `usePlanStore((s) => s.optimisticEvents)` 订阅读取
  （`MonthCalendarView.tsx:108`），chip / grid 均不调用任何 setter。无 silent catch /
  fallback；无 patch 时为 no-op spread（保留原 read-only 行为，注释明示 AGENT-H7）。✅
- key 一致性：`optimisticEvents.get(item.event.id)`（`MonthCalendarView.tsx:165`）。store
  key 类型为 `ReadonlyMap<string, OptimisticEventPatch>`，`OptimisticEventPatch =
  Partial<PlanEventReadV2>`（`usePlanStore.ts:12,19`），`event.id: string`。key 与 store
  string key 完全一致。✅
