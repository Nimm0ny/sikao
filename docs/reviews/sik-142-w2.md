## Findings

### Initial Findings (Resolved Before Pass)

1. `F1` `Medium` — Sunday-first month cells and query window were misaligned.
   Evidence:
   - `packages/calendar-engine/src/view.ts` originally fixed `month` to Monday-start while `MonthCalendarView.tsx` still honored `config.startWeekOnMonday`.
   - This caused Sunday-first grids to render `2026-05-24` while the fetch window started at `2026-05-25`.
   Resolution:
   - `ViewRangeAnchor` now accepts `startWeekOnMonday`.
   - `buildViewRange()` uses that flag for both `week` and `month`.
   - `MonthCalendarView.tsx` and `WeekCalendarView.tsx` pass the config flag through.

2. `F2` `Low` — W2 tests were too weak to catch the Sunday-first regression.
   Evidence:
   - `MonthCalendarView.test.tsx` only asserted that the first two DOW labels differed, which could pass on incorrect orderings.
   - Error-state coverage no longer locked the `重试` action.
   Resolution:
   - Sunday-first test now asserts exact labels, exact 21-cell span, and a `2026-05-24` event landing in the visible cell.
   - Error-state test again asserts the `重试` button.

### Open Findings

- None.

## Residual Risk

- Low: `MonthGridDnd` does not yet have a dedicated Sunday-first drag/conflict test. The runtime window passed into DnD now shares the same `buildViewRange(..., startWeekOnMonday)` path, so behavior is inferred consistent, but a future W3/W4 pass should add an explicit DnD coverage case.
