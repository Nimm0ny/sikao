# Phase-Home · 01 · Boundary Rules

> **Status**: ACCEPTED
> **Last Updated**: 2026-05-21
> **Index**: see `./README.md`

本文是首页"计划 / 推荐 / 实绩"三层之间的边界规则 SSOT。所有后端服务、LLM prompt、前端 UI 必须遵守。

---

## 1. 六条核心规则（P1-P6）

### P1 学习计划 = 目标 + 路径建议

- **目标层**（提分 / 通关）才是计划的本质。
- **路径层**（日历事件）只是建议，不是计划成败的衡量标准。
- 数据：目标层 = `PlanV2`（含考试日 / 每日时长 / 起点 / 风格 / 重点科目 / 多 exam_targets）；路径层 = `PlanEventV2[]`。
- UI 含义：首页 Section A 的"日历画廊"显示的是路径层；目标层在画廊上方一行 chip + 倒数考试日。

### P2 实绩层独立于计划层

- 所有 `PracticeSessionV2` 都贡献进度，无论是否绑定计划事件。
- 对应字段：`PracticeSessionV2.linked_plan_event_id` 可为空。
- ProgressSnapshot 与 WeaknessSnapshot 的写入器**只读 PracticeSessionV2 + EssaySubmission/Report**，**绝不读 PlanEventV2.status**。

### P3 PlanEventV2.status 只表达事件本身

- `done` / `skipped` 只表达"这个时间块是否被执行了"，不代表"今日言语任务成功了"。
- 用户在事件外的额外 session 不会让事件 status 改为 done。
- "今日完成度" UI 指标 = 实绩层聚合（按今日时间窗口里的 session 计算），不是事件 status 计数。

### P4 用户额外练习自动落入日历视图

- unlinked session（不来自计划事件 CTA）也要在日历上显示，但**作为"实绩块"而非"事件块"**。
- 数据：通过 `GET /events?include_practice_blocks=true` 联合返回。响应 shape 见 `02-Data-Model.md` §6.1。
- 视觉：实绩块用半透明虚线边框 + 不同 chip（"已完成 / 已答 X 题"），不可拖拽编辑。

### P5 AI 推荐基于实绩 + 目标 + 实时状态

- 输入信号：实绩聚合、目标差距、当前疲劳度（连续答题时长）、近期错误密度、近期错题积压。
- AI 推荐**永远不读** PlanEventV2.status。
- 推荐刷新触发：见 `00-Decisions.md` Rec-7 + Infra-Cron + session.submit hook。

### P6 任何"改计划"的行为都需审计 + 提案制

- **用户提案**：用户编辑 / 创建 / 删除事件 → 直接写 `PlanEventV2` + `PlanEventV2.change_log` 追加一条。
- **AI 提案**：AI 调整 → 写 `PlanAdjustmentV2`（pending），用户 accept 后才写 `PlanEventV2`。
- **AI 制定**：用户主动触发 → 直接生成 `PlanEventV2[]`（不走 adjustment 流），但 `PlanV2.change_log` 记一条"ai_generated by AI-1 trigger"。
- 所有变更必须有 actor / before / after / reason / created_at。

## 2. 推荐策略阈值表（Infra-Rec-Policy）

存放路径：`services/api/src/sikao_api/modules/llm/application/llm/recommender_policy.py`

### 2.1 信号 → 推荐映射

| 信号 | 阈值 | 推荐动作 |
|---|---|---|
| 当前正确率 | < 60% | **复盘**（不继续堆量） |
| 当前正确率 | 60-80% | 复盘 OR 巩固同题型 |
| 当前正确率 | > 80% | 继续推进（换新题型 / 升级难度） |
| 累积错题数 | > 30 题未复盘 | **复盘**优先于继续 |
| 连续答题时长 | > 60 分钟 | 提示休息（不强制） |
| 连续答题时长 | > 90 分钟 | **复盘**或休息（不再推新题） |
| 距考试日 | < 30 天 | 模考权重 ↑ |
| 距考试日 | < 7 天 | 几乎只推复盘 + 模考 |

### 2.2 信号窗口定义

| 信号 | 计算窗口 | 数据源 |
|---|---|---|
| 当前正确率 | 最近 50 题（不限时间） | PracticeSessionV2.items |
| 累积错题数（未复盘） | 全部历史 | review.items WHERE reviewed_at IS NULL |
| 连续答题时长 | 当前未提交 session 的 elapsed | 实时 session state |
| 距考试日 | 当前日期 与 ProfileGoal.exam_targets[] 中最近未来日期之差 | profile_v2 |
| 错误率上升趋势 | 最近 10 题 vs 之前 50 题正确率差 | session.items |

### 2.3 阈值表用法

阈值表既作为：
1. **LLM prompt 上下文**（让 AI 理解策略，详见 `06-LLM-Prompts.md` §3）
2. **后端硬约束**（recommender 输出后用阈值表二次校验，违反阈值的推荐被丢弃 → fallback 到第二候选）

### 2.4 用户可调（Stage 2 预留）

`profile_v2.info.recommender_preferences` JSON 字段预留：

```json
{
  "fatigue_threshold_minutes": 60,
  "review_priority_when_accuracy_below": 60,
  "exam_proximity_days_aggressive": 30
}
```

Stage 1 不暴露 UI；用户改值要直接 PUT 接口（开发者用）。

## 3. 场景示例

### 3.1 场景 A：用户加练后 AI 如何响应

**场景**：用户今日计划事件"30 题言语"已 done，正确率 50%。用户主动又做了 30 题（unlinked session，非计划事件触发）。

**系统行为**：

1. **实绩层（P2）**：60 题全部计入进度，无论是否绑定计划事件。
2. **日历视图（P4）**：计划事件块显示"✓ done"；额外 30 题作为"实绩块"显示在同一时段，视觉区分。
3. **进度计算（P3）**：今日言语正确率 = 60 题聚合结果（不只看计划内 30 题）。
4. **AI 推荐触发（P5）**：session.submit 后实时调 recommender。

**AI 推荐决策树**：

```
输入信号：
├─ 计划层：今日言语事件已 done（30 题）
├─ 实绩层：今日已答 60 题言语，正确率 50%（差目标 25 个百分点）
├─ 实时状态：连续答题约 90 分钟，最近 10 题错 6 题（错误率上升）
├─ 历史：近 7 天言语类累积错题 38 题未复盘
└─ 目标差距：言语目标正确率 75%，当前 50%，差 25 个百分点

→ 阈值匹配（§2.1）：
  - 正确率 50% < 60%  → 触发"复盘"
  - 累积错题 38 > 30  → 触发"复盘优先"
  - 连续答题 90 分钟 > 90 → 触发"复盘或休息"
  - 错误率上升趋势   → 强化"复盘"权重

→ AI 输出推荐（按优先级排序）：
  1. [复盘] "复盘 5 题逻辑判断（10 分钟）"
     原因："你最近言语错题 38 题未复盘，今日正确率 50%，继续堆量效果不好"
  2. [休息] "建议休息 15 分钟"
     原因："已连续学习 90 分钟，错误率在上升"
  3. [继续] "换子题型：推断推理 10 题"
     原因："避免在同一弱项上无效堆题，换一个切入点"
```

**核心原则**：
- 正确率 < 60% 时 → **优先推复盘**
- 错误率呈上升趋势 → **不再推新题**
- 推荐永远显示原因（Rec-3）
- 推荐不直接改计划（P6）；用户接受后选"进 session"或"加入明天计划"

---

### 3.2 场景 B：跳过事件触发 AI 调整

**场景**：用户今日 19:00-20:00 计划事件"30 题判断推理"未做，22:00 用户登录。

**系统行为**：

1. **事件状态**：cron 每天 00:00 把过期未做的事件 status 自动改为 `skipped`。
2. **触发 AI 调整（ADJ-1）**：cron 写完 skipped 后立即调 plan_adjustor。
3. **限流（ADJ-6）**：检查近 24h 是否已 reject 过类似提案，若是则跳过。
4. **生成 PlanAdjustmentV2**（pending），写 reason = "近 3 天判断推理跳过 2 次，建议把 30 题分散到本周末上午"。
5. **用户登录**：banner 弹出，用户 accept → 落地到 PlanEventV2，写 change_log；reject → adjustment 标记 rejected + reason。

---

### 3.3 场景 C：AI 制定计划遇到考试日已过

**场景**：用户填表单时填了过去的考试日（误操作）。

**系统行为**：

1. **前端校验**：考试日 input 设置 `min={today}`，禁止选过去日期。
2. **后端校验**：`POST /plans/auto-generate` schema 校验 `target_exam_date >= today + 1 day`，否则返回 422。
3. **LLM 永远收不到无效输入**（fail-fast）。

---

### 3.4 场景 D：unlinked session 与 linked session 同时存在

**场景**：今日计划事件"19:00-20:00 行测 30 题"，用户 19:00 进入事件 CTA 开始 session（linked），但 19:30 中途交卷只做了 15 题；19:45 用户又主动开了一个 session 做 20 题（unlinked）。

**系统行为**：

| 时刻 | 状态 |
|---|---|
| 19:00 | session_1 创建 (linked_plan_event_id=event_x)，event status = in_progress |
| 19:30 | session_1.submit，做了 15 题，event status 仍 = in_progress（用户没显式标完成） |
| 19:45 | session_2 创建 (linked_plan_event_id=null)，事件块继续显示 in_progress |
| 19:55 | session_2.submit，做了 20 题 |
| 20:00 | event 时间窗口结束，event status 由 cron 改为 done（因为 linked session 存在 + 在时间窗口内有 session 提交） |

**事件 status 机器规则**（见 `02-Data-Model.md` §3.2）：

```
planned ──[time enter window]──→ in_progress
in_progress ──[linked session submit]──→ in_progress (still, until time window ends)
in_progress ──[time window ends + ≥1 linked session submitted]──→ done
in_progress ──[time window ends + 0 linked session]──→ skipped
```

> 注意：unlinked session 不影响 event status（P3）。它只贡献进度（P2）和触发推荐（P5）。

## 4. "仍在学习计划内"的精确语义

用户加练（unlinked session）**不会**被排除在学习计划体系之外。

| 维度 | 计划内 session（linked） | 加练 session（unlinked） |
|---|---|---|
| 贡献进度 | ✅ | ✅（因 P2） |
| 影响 AI 推荐 | ✅ | ✅（因 P5） |
| 触发 AI 调整提案 | ✅ | ✅（基于实绩，不是事件 status） |
| 显示在日历 | 作为计划事件块 | 作为实绩块（因 P4） |
| 改变计划事件 status | 是（planned → done） | 否 |
| 进入学情分析 | ✅ | ✅ |

**一句话**：学习计划的度量标准是**目标达成（提分）**，不是**事件完成率**。用户无论通过计划事件还是自由加练获得的实绩，都 100% 贡献到目标进度和 AI 决策中。

## 5. 模糊点的精确判定（前文遗漏的 case）

### 5.1 推荐 accept 进 session 时的 link 双绑定

| Rec-9 决策 | 实现 |
|---|---|
| `linked_recommendation_id` 字段写到 PracticeSessionV2 | 是 |
| 同时是否写 `linked_plan_event_id`？ | **否**。推荐 accept 走"进 session"分支时，session 只绑 recommendation；走"加入计划"分支时，先创建 PlanEventV2，再如果用户立即从该事件 CTA 进入，才绑 event |

### 5.2 plan_id 切片在进度上的语义

`GET /dashboard/progress?plan_id=X`：

- **plan_id 不影响实绩聚合的 session 集合**（P2/P3：进度独立于计划）
- plan_id 只影响"目标值"（progress.summary 中的 target_xxx 字段从该 plan 取，而不是 user 的最近 plan）
- 即同一组 session，按不同 plan 切片返回的"差距值"不同，但"已达成值"相同

### 5.3 "倒数考试日"按哪个 target 计算

- 多 exam_targets 时，倒数显示**最近未来的那个**。
- 若过去日期出现在 exam_targets 中（用户考完没删），自动忽略，取下一个未来。
- 全部都是过去 → 显示 "无未来考试日"，CTA "前往设置 →"。

### 5.4 unlinked session 的实绩块时间窗口

- 实绩块的 start = session.started_at，end = session.submitted_at。
- 若 session 还在 in_progress（未提交），实绩块显示 start ~ now（实时延伸），刷新页面会更新。
- 实绩块跨越多日（罕见，比如用户没交卷过夜）时按多日事件渲染（Cal-3 同样规则）。

### 5.5 重复事件展开后的 session 绑定语义

- RRULE 序列里的每一次实例都有独立 `recurring_parent_id`，但**它们不是 DB 里的独立行**（虚拟实例）。
- 用户从某次实例的 CTA 进入 session 时：
  - `linked_plan_event_id` 写母规则 event 的整数 FK（`PlanEventV2.id`）
  - `linked_plan_event_occurrence_ref` 写该次实例的展开 ID（格式 `<parent_id>:<occurrence_iso_date>`）
- 虚拟实例的 session 关联以 `PracticeSessionV2(linked_plan_event_id, linked_plan_event_occurrence_ref)` 为权威来源；`plans/events` 响应里的 `linked_session_id` 应按这对键动态推导，不复用父规则行上的单值 shortcut。
- detached 单次例外进入 session 时，不写 `linked_plan_event_occurrence_ref`；`linked_plan_event_id` 直接指向 detached event 行。
- 数据模型上，DB 只存 `parent` 与 `exception_dates[]` 与 `single_overrides[event_id_list]`。详见 `02-Data-Model.md` §3.4。

### 5.6 软删除事件后，已 linked 的 session 怎么办

- session 不删（hard），保留 `linked_plan_event_id` 指向已软删的事件。
- 若该 session 来自 recurring 虚拟实例，`linked_plan_event_occurrence_ref` 也继续保留，作为历史定位信息。
- 查询 session 详情时如果引用的事件已软删，返回 event 字段时附 `is_deleted: true`。
- 进度聚合不受影响（实绩层独立）。
